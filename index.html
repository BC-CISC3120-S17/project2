Project Two: WebSockets Checkers

Overview

The main objective of this project is to give you experience writing WebSockets code.

The project is due at 8am on the day of the final. I will clone your repositories for grading at that time.

The Project

In this project, you will take the infrastructure for a simple GUI game—checkers—and add WebSockets code that will allow the game to be played over a network.

The Existing Code

The code in this repository is in three packages.

checkers.client contains GUI code for drawing a checkerboard. CheckersClient is a simple Swing application which draws a checkerboard and also listens for mouse activity. CheckersCanvas is the class that actually draws the board—it also includes code that helps the controller convert mouse-clock locations into board locations. The CheckersCanvas includes a reference to a CheckerBoard — see below.

checkers.common contains classes used by both the client and server. These provide part of the "model": it represents the board (CheckerBoard), the squares on the board (Square), and the players who may occupy those squares (SquarePlayer). The client, for example, will use this lightweight model to draw the board.

Finally, checkers.server contains a "heavier" model (CheckersModel) which includes the rules of checkers— which jumps are legal? when does a checker get kinged? and so on. Note that the client will run as it is, but no checkers will be drawn.

A Network Architecture

This code is ready to be embedded in a WebSockets application. Here's the idea: Once a "checkers server" starts, clients can connect in order to play. Once two clients have connected, the server lets them start playing. When a client tries to make a move, a message is sent to the server. If the move is not legal, an "illegal move" message is sent back to the client, and nothing else happens. If the move is legal, then the server sends a message to both clients, letting them know that the move has been made and indicating that it's the other player's turn. The clients, in turn, should update their display of the board and let the users know whose turn it is.

Important: when a user tries to make a move, the client should not display that move immediately—it MUST send the move to the server. (The client doesn't have enough intelligence to know whether the move is legal, or even whether it's the local player's turn to move.) The display should only ever be updated in response to a message from the server.

Of course, when a player wins, the server should send an appropriate message, and the clients, upon receiving that message, should end play.

What You Need to Do

The GUI code and checkers-model code is in good shape. Your focus is on communication: What kinds of messages will the clients send to the server? What kinds of messages will the server send to the clients?

How will your client code actually send messages to, and receive messages from, the server? When should such messages be sent?

On the server side, how will you code a relationship between the WebSockets endpoint and the checkers model?

Don't worry! We're going to look at a few other WebSockets applications so that you'll have some examples to build from.

Behavior

Your application should

  1. allow clients to connect to the server
  2. allow the clients to play a "legal" game of checkers
  3. inform players when a game has ended
